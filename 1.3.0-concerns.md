
Here is an audit focused on **safety** (operational / airspace-style constraints), **security** (abuse and trust boundaries), and **stability** (robustness under failure). It is based on the current `aerpawlib` v1/v2 code paths you have in the repo.

---

## Executive summary

| Area | v1 | v2 |
|------|----|----|
| **Safety** | Geofence/speed/alt logic exists in `SafetyCheckerServer`, but **v1 CLI does not wire it into `Vehicle`**. Mission code must call the client explicitly. | `can_takeoff` / `can_goto` / `can_land` combine local checks + optional `SafetyCheckerClient`, but **`takeoff` / `goto_coordinates` / `land` do not call them** — checks are opt-in. `NoOpSafetyChecker` **always passes** when the server is missing (non-AERPAW without port). |

## Safety (operational constraints)

**What works well**

- **Shared safety server** (`aerpawlib.v1.safety.server.SafetyCheckerServer`): waypoint path checks against include/exclude polygons, altitude band for copters, speed limits, takeoff altitude, and landing within **5 m of recorded takeoff** (server-side state). Config is loaded with `yaml.safe_load`.
- **v2 local preconditions** in `can_*`: armable, 3D GPS, battery threshold, then optional async safety client.
- **v2** documents and implements **unexpected disarm** handling in the runner layer (`_run_with_disarm_guard` in `runner_impl.py`), which is a useful stability/safety signal for “vehicle dropped out of the mission unexpectedly.”

**Gaps and risks**

1. **Safety checks are not enforced by movement APIs (v2)**  
   `Vehicle` exposes `can_takeoff`, `can_goto`, `can_land` in `base.py`, but `Drone.takeoff`, `Drone.goto_coordinates`, `Drone.land` in `drone.py` go straight to MAVSDK after `await_ready_to_move()` — they **do not** await `can_*`. So geofence/speed validation is **manual**: callers must remember to check first. The CLI injects a safety client into `connect(..., safety=...)`, but that only helps if every mission uses `can_*` before commands.

2. **`NoOpSafetyChecker` explicitly passes everything** (`checker.py`): intentional for degraded mode, but operationally dangerous if operators assume “safety is on” because v2 was started with a port that failed — the code falls back and logs.

3. **v1 experiments never attach `SafetyCheckerClient` to `Vehicle`** (`cli/experiments_v1.py`): unlike v2, there is no automatic injection; the safety stack is a **library** unless user scripts use it.

4. **PreflightChecks** (`v2/safety/validation.py`) is **not called** from runners or vehicle connect in-repo — it is available but not part of the default lifecycle.

5. **Server-side landing rule** depends on **`validate_takeoff` having run** so `takeoff_location` is set; otherwise landing validation fails with a clear message — good, but order-dependent.

6. **Rover vs copter**: copter gets altitude checks; rover path still uses polygon logic; landing distance check applies whenever the landing handler is used (handler doesn’t branch on `vehicle_type` for landing — worth confirming for rover missions).

## Security (trust boundaries)

**Critical / high**

1. **ZMQ `recv_pyobj` / `send_pyobj`** (`v1/runner_impl.py` `ZmqStateMachine`, `v2/runner_impl.py` `ZmqStateMachine`): this uses **pickle under the hood**. Anyone who can publish to the proxy’s input side can deliver **arbitrary object deserialization** to subscribers — a classic **remote code execution** pattern if the ZMQ proxy listens on non-loopback interfaces or an untrusted machine can reach it. **Treat the ZMQ proxy + runners as a trusted enclave; bind only on `127.0.0.1` or use firewall rules; do not expose across hostile networks.**

2. **Safety checker server** (`v1/safety/server.py`): binds `tcp://*:{port}` — **all interfaces**, **no authentication**. Any host that can reach the port can send requests. Wire format is zlib-compressed JSON with **no maximum message size** before `zlib.decompress` — **ZIP/decompression bomb** and **CPU exhaustion** risk.

3. **v1 `ExternalProcess.start`** (`v1/external.py`): builds a **single shell string** (`executable + params + shell redirects`) and uses `asyncio.create_subprocess_shell`. If `params` or paths ever come from config/network, this is **command injection**. v2 switched to `create_subprocess_exec` with an argv list — **prefer v2’s pattern** and treat v1 as unsafe for untrusted input.

**Medium / lower**

- **Error detail in safety server responses**: generic handler returns `Server error: {e}` to the client — can leak internal exception text to whoever sent the message (minor information disclosure on a LAN).
- **JSON in waypoint handler**: client-controlled JSON strings parsed with `json.loads` — no structural validation beyond `Coordinate` construction; malformed types could raise (caught) but **large strings** are still a DoS vector.

## Stability (robustness)

**Improvements in v2**

- **State machine background tasks**: capped retries with exponential backoff (`MAX_BACKGROUND_RETRIES = 10` in `v2/runner_impl.py`) vs v1’s **infinite restart loop** on any exception (`v1/runner_impl.py` `_task_runner`).
- **ZmqStateMachine**: v2 processes messages **sequentially**, uses **`asyncio.Event`** for field queries instead of **0.01s busy-poll** (v1 `query_field`), and **destroys ZMQ context** in `finally` after `run`.
- **Safety client**: v1 sync ZMQ with `RCVTIMEO`/`SNDTIMEO`; v2 async client uses `asyncio.wait_for` on send/recv — both avoid **indefinite hang**; v2 also validates presence of `result` in the response (`checker.py`).
- **ConnectionHandler** (`v2/safety/connection.py`): heartbeat timeout and optional disconnect callback; **SIGINT/SIGTERM** via `add_signal_handler` where supported.

**Remaining stability concerns**

- **v1 ZMQ SUB loop**: `recv_pyobj` failure modes — if deserialization raises, behavior depends on exception handling (v2 recv loop logs and continues).
- **Safety server loop**: single-threaded REQ handler — one slow validation blocks all clients (acceptable for small deployments, **DoS** if abused).
- **v2 `SafetyCheckerClient`**: broad `except Exception` on send/recv triggers reconnect — good for REQ state recovery, but callers must still handle failures.

---

## Recommendations (prioritized)

1. **Treat ZMQ pickle as unsafe on any routable network** — document, firewall, or replace with JSON/msgpack and schema validation for remote control messages.
2. **Bind safety server to loopback** (`127.0.0.1`) or add auth/TLS if it must be remote; add **max compressed/uncompressed size** before decompress.
3. **Either** call `can_takeoff` / `can_goto` / `can_land` from the actual command methods **or** rename/document them as **mandatory preconditions** so missions cannot skip them by accident.
4. **Deprecate or harden v1 `ExternalProcess`** — migrate callers to v2-style `exec` + explicit file handling.
5. **Make `NoOpSafetyChecker` mode very visible** (e.g. startup banner, non-zero exit in strict profiles) for anything beyond local SITL.
